# Runtime notes

- The skin launches the official Codex desktop app with `--remote-debugging-port=<port>` and injects through CDP. Windows resolves the Store `ChatGPT.exe`; macOS resolves `CFBundleExecutable` from `ChatGPT.app` / `Codex.app` and launches it through LaunchServices (`open -na`).
- Themes are scanned at injector start from `themes/` and `themes-private/` (folders containing `theme.json`; see `THEME-SPEC.md`). Adding or editing a theme requires re-running the platform `start-dream-skin` script so the daemon rebuilds its payload; the running renderer keeps the old payload until then. `node scripts/injector.mjs --themes` prints the scan result without touching the app.
- Invalid themes are skipped (and unscoped `extra.css` files rejected) with `[dream-skin]` warnings on stderr. Check `injector-error.log` under `%LOCALAPPDATA%\CodexDreamSkin` (Windows) or `~/Library/Application Support/CodexDreamSkin` (macOS) when a theme silently fails to appear.
- Theme/layout switching is programmatic only (`scripts/set-theme.mjs`); the selection persists in the renderer's localStorage under `codex-dream-skin.theme` / `codex-dream-skin.layout` and survives reloads, restore/reapply cycles, and watcher-recovered restarts. The default theme comes from `"default": true` in a theme.json (fallback: first scanned theme); the default layout is `fullscreen`.
- The default production port is `9335`; test instances may use another port plus an isolated `--user-data-dir`. On macOS, `install-state.json` remembers an installer-selected port and app path so later unified commands use the same values automatically.
- CDP is bound to loopback. Do not expose it on a LAN interface.
- **Loopback stack is not stable across reboots**: Chromium may bind the DevTools server to `127.0.0.1` or to `[::1]` only (observed switching after a Windows reboot on 2026-07-15; hosts-file localhost entries do not decide it). Every probe — injector, launcher, watcher — must try both `http://127.0.0.1:<port>/json/list` and `http://[::1]:<port>/json/list` and accept either. Hardcoding IPv4 caused a watcher kill-loop that looked like Codex "闪退" while the app was actually healthy on `[::1]`.
- The watcher has a circuit breaker: after `MaxConsecutiveFailures` (default 3) failed recoveries it stops touching Codex for `CooldownMinutes` (default 30), logging the suspension. Codex keeps running unskinned during cooldown; a healthy skin state (CDP + injector) resets the counter automatically. The watcher must NEVER kill-loop Codex — any change to recovery logic must preserve this property.
- **Single probes are not trustworthy**: while Codex boots (session restore, window churn), `/json/list` can transiently return no `app://` page or time out even though the skin is fine. Observed on 2026-07-15: a freshly recovered instance was re-killed 6–15 s after a successful verification because one probe flaked, producing a "successful-restart loop" that the failure-based breaker cannot catch. Two watcher defenses exist and must both stay: (1) debounce — `ProbeFailuresBeforeRecovery` (default 3) consecutive missed probes are required before any recovery, with a 3 s HTTP timeout per probe and `LaunchGraceSeconds` (default 15) of leniency for young processes; (2) restart rate limit — at most `MaxRestartsPerWindow` (default 2) Codex restarts per `RestartWindowMinutes` (default 10), beyond which auto-recovery suspends for `CooldownMinutes` even if every restart "succeeded".
- After force-killing `ChatGPT.exe`, Windows may auto-respawn it without the debug flag; the respawn can win the single-instance race and silently drop CDP. The Windows launcher waits for the process list to actually drain, swats late respawns, and retries the flagged launch once before giving up.
- On macOS, starting `Contents/MacOS/ChatGPT` directly can expose CDP briefly and then be reaped by the application lifecycle. Always launch the bundle through LaunchServices. Current Codex macOS builds also close the renderer CDP socket on `Page.captureScreenshot`; `verify-dream-skin.sh` therefore resolves the owning process and Quartz window ID, then calls the native `screencapture -l<window-id>` path.
- Current official macOS Codex bundles include `Contents/Resources/cua_node/bin/node` (Node.js 24 in the tested build). macOS scripts prefer a compatible system Node.js when present and otherwise use this bundled runtime, so ordinary users do not need a separate Node installation. Keep explicit `--node` support for testing and future bundle changes.
- The macOS installer atomically syncs `scripts/`, `assets/`, `styles/`, and built-in `themes/` into `~/Library/Application Support/CodexDreamSkin/runtime`. Private themes live durably at `~/Library/Application Support/CodexDreamSkin/themes-private`; the runtime links to that directory so generated themes survive runtime replacement. The LaunchAgent and post-install commands must point at the stable runtime, never the downloaded repo. A reinstall must stop old watcher/injector processes before replacing it.
- macOS quick-theme uses the system `/usr/bin/sips` command to decode and resize PNG/JPG input to a 64×64, uncompressed BMP sample. `generate-quick-theme-macos.mjs` reads those pixels, applies the same quantized-cluster/HSL derivation as `quick-theme.ps1`, emits all 28 required tokens, and refuses to overwrite built-in or manually-authored themes. No third-party image package is required.
- The injector polls page targets and reinjects after document loads. In-page route changes use a debounced observer plus a low-frequency safety check to avoid CPU churn during streamed tasks.
- Target selection is intentional: only the main `app://-/index.html` page receives the skin. Compact and `initialRoute` pages are treated as auxiliary windows; the daemon removes stale Dream Skin state from them so desktop-pet transparency survives upgrades, reloads, and prior buggy injections.
- `state.json` records the port and daemon PID. On macOS, `install-state.json` separately records durable install defaults and is removed only by full uninstall. State/log roots are `%LOCALAPPDATA%\CodexDreamSkin` on Windows and `~/Library/Application Support/CodexDreamSkin` on macOS.
- Windows installs `Codex Dream Skin Watcher.lnk` in Startup. macOS installs `~/Library/LaunchAgents/com.codex-autoskin.watcher.plist`. The watcher does not launch Codex by itself; it only reacts when Codex is opened normally without the required debug port.
- A newly loaded macOS LaunchAgent ignores an already-running unskinned Codex instance until that instance closes or is explicitly relaunched with CDP. Installation must never silently terminate the user's current work.
- `watcher-state.json` records the watcher PID and `watcher.log` records recovery events. Windows uses a named mutex; macOS uses an atomic lock directory with stale-PID recovery.
- If Codex is already running without the chosen debugging port, close it first or explicitly use `-RestartExisting`.
- App updates are supported because Windows queries `Get-AppxPackage OpenAI.Codex` on every launch and macOS re-reads the selected bundle's `Info.plist`.
